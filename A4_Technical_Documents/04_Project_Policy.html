2. Git Branching Strategy

The project uses a simplified Git Flow model with two long-lived branches and short-lived feature/fix branches.

main ─────────────────────────────────────▶ (stable, production-ready) ↑ merge via PR (releases only) develop ──────────────────────────────────────▶ (integration branch) ↑ merge via PR after review feature/* ──────────▶ (new features, short-lived) fix/* ──────────▶ (bug fixes, short-lived)

2.3 Feature Branch Workflow

# 1. Pull latest develop
git checkout develop
git pull origin develop

# 2. Create feature branch
git checkout -b feature/your-feature-name

# 3. Implement and commit (small, atomic commits)
git add src/controllers/campaigns.controller.js
git commit -m "feat: add campaign creation endpoint"

# 4. Push and open Pull Request → develop
git push origin feature/your-feature-name
#   → Open PR on GitHub: feature/... → develop

# 5. After PR approved and merged, delete the branch
git branch -d feature/your-feature-name

3. Code Review Process

3.1 Pull Request Rules

RuleDetail
Minimum reviewers1 — ideally the team member most familiar with the affected module
PR sizeTarget < 400 changed lines. Break large features into smaller PRs
PR descriptionMust state: what changed, why it changed, and how to test it
Self-reviewAuthor reviews their own diff before requesting review
No force-pushdevelop and main are protected — no git push --force
Tests requiredNew endpoints must include or update test cases in tests/
Lint must passnpm run lint must exit with 0 errors before marking PR ready

3.2 Reviewer Checklist

  • Code follows the Airbnb style guide (ESLint passes)
  • New routes are protected with the correct auth + role middleware
  • Inputs are validated using express-validator
  • Errors are handled and passed to next(err)
  • No N+1 Sequelize queries (check depth of include)
  • No hard-coded secrets — all env values read from process.env
  • New environment variables are documented in .env.example
  • Tests cover the happy path and at least one error case

5. Secrets & Environment Policy

Critical: .env files must never be committed to the repository. All sensitive values must be stored in environment variables.
RuleDetail
.env never committed.gitignore excludes .env, .env.local, .env.production
.env.example always updatedAny new variable must be documented with a placeholder value
Production secretsStored in the host's env manager (Railway, Heroku) — never hardcoded
Different credentialsDevelopment and production databases must use separate credentials

6. Definition of Done

A task is complete when all of the following are true:

  1. Feature is implemented and works locally
  2. ESLint passes: npm run lint exits with 0 errors
  3. Tests are written and pass: npm test all green
  4. PR is opened, reviewed, and approved by at least one team member
  5. PR is merged into develop
  6. Documentation (API docs, README, .env.example) is updated if needed
